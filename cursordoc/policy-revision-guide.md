# 政策修订功能使用指南

## 📋 功能概述

政策修订功能允许您对已发布（已批准）的政策进行更新，通过创建新版本而不是直接修改原政策来维护政策的完整性和审计轨迹。

## 🔄 修订工作流程

### 1. 创建政策修订

**适用场景：**
- 政策内容需要更新
- 政策适用范围发生变化
- 政策有效期需要调整
- 政策负责人变更

**操作步骤：**
1. 在政策列表中，找到需要修订的已批准政策
2. 点击"创建修订"按钮
3. 填写修订原因（必填）
4. 更新政策信息（名称、摘要、部门等）
5. 上传新的政策文档（可选）
6. 提交修订申请

### 2. 修订审批流程

**审批状态：**
- `pending` - 等待审批
- `approved` - 修订已批准
- `rejected` - 修订被拒绝

**审批权限：**
- 部门负责人审批本部门政策修订
- 超级用户可以审批所有政策修订

### 3. 政策版本管理

**版本状态：**
- `active` - 当前生效的政策版本
- `superseded` - 已被新版本替代的政策
- `draft` - 草稿状态
- `pending` - 等待审批

## 🛠️ 技术实现

### 后端API端点

#### 创建政策修订
```http
POST /api/v1/policy-revisions/{policy_id}/create-revision
Content-Type: multipart/form-data

{
  "policy_name": "Updated Policy Name",
  "policy_summary": "Updated summary",
  "revision_reason": "Reason for revision",
  "file": "policy_document.pdf"
}
```

#### 获取政策修订历史
```http
GET /api/v1/policy-revisions/{policy_id}/revisions
```

#### 获取版本历史
```http
GET /api/v1/policy-revisions/{policy_id}/version-history
```

#### 替代政策
```http
POST /api/v1/policy-revisions/{policy_id}/supersede
Content-Type: application/json

{
  "superseding_policy_id": 123
}
```

### 数据库模型更新

**Policy表新增字段：**
- `parent_policy_id` - 父政策ID（用于链接修订）
- `revision_reason` - 修订原因
- `superseded_by_id` - 被替代的政策ID
- `version_number` - 版本号

### 前端组件

**PolicyRevisionForm组件：**
- 政策修订表单
- 支持文件上传
- 表单验证
- 错误处理

## 📊 使用示例

### 场景1：更新员工手册

1. **原始政策：** "员工手册 v1.0" (已批准)
2. **修订原因：** "更新休假政策，增加远程工作条款"
3. **新版本：** "员工手册 v2.0" (待审批)
4. **审批后：** v1.0 状态变为 "superseded"，v2.0 变为 "active"

### 场景2：调整报销政策

1. **原始政策：** "差旅报销政策" (已批准)
2. **修订原因：** "提高住宿标准，增加绿色出行奖励"
3. **新版本：** "差旅报销政策 v2.0" (待审批)
4. **审批后：** 新版本生效，旧版本归档

## 🔒 权限控制

### 修订权限
- **政策创建者：** 可以创建自己政策的修订
- **超级用户：** 可以创建任何政策的修订
- **部门负责人：** 可以审批本部门政策的修订

### 访问权限
- **普通用户：** 只能查看当前生效的政策
- **政策创建者：** 可以查看自己政策的所有版本
- **审批者：** 可以查看待审批的修订

## 📈 审计和追踪

### 修订历史记录
- 每次修订都有完整的审计轨迹
- 记录修订原因、时间、操作人
- 保留所有版本的政策文档

### 版本比较
- 可以比较不同版本的政策内容
- 显示变更摘要
- 追踪政策演进过程

## 🚀 最佳实践

### 修订政策时
1. **明确修订原因：** 详细说明为什么需要修订
2. **保持一致性：** 确保修订内容与组织目标一致
3. **及时沟通：** 通知相关人员政策变更
4. **文档更新：** 确保相关文档同步更新

### 审批修订时
1. **仔细审查：** 检查修订内容的合理性和完整性
2. **考虑影响：** 评估修订对组织的影响
3. **及时处理：** 避免审批流程过长
4. **记录意见：** 在审批意见中说明决策理由

## 🔧 故障排除

### 常见问题

**Q: 为什么无法创建修订？**
A: 检查政策状态是否为"approved"，只有已批准的政策才能创建修订。

**Q: 修订审批后原政策状态如何？**
A: 原政策状态会变为"superseded"，新版本变为"active"。

**Q: 如何查看政策的完整历史？**
A: 使用版本历史API或前端版本历史页面查看所有版本。

**Q: 可以撤销已批准的修订吗？**
A: 不能直接撤销，但可以创建新的修订来纠正问题。
