## 二、功能详细说明（面向客户）

### 功能1：工作流审批异步处理 ⭐⭐⭐⭐⭐
**功能描述**：
- 工作流执行到审批节点时，系统不再阻塞等待，而是暂停执行并释放服务器资源
- 审批人可以在任意时间完成审批，系统会自动恢复工作流执行
- 支持多个工作流同时运行，互不干扰

**实际应用场景**：
1. **批量文档审批**：财务部门需要审批100份报销单，系统可以同时处理多个审批流程，不会因为某个审批人不在线而阻塞其他流程
2. **跨时区协作**：美国团队提交的工作流需要中国团队审批，系统可以在审批人上线时自动恢复执行
3. **高并发场景**：销售团队同时提交50个合同审批流程，系统可以并行处理，不会因为资源阻塞导致系统卡顿

**带来的价值**：
- ✅ **性能提升**：服务器资源利用率提升80%，支持10倍以上的并发工作流
- ✅ **用户体验**：审批响应速度提升，不再出现"系统卡顿"的情况
- ✅ **成本节约**：相同硬件配置下可以处理更多业务，降低服务器成本

**技术实现**：使用LangGraph的interrupt机制，替代当前的轮询方案

---

### 功能2：工作流模板保存 ⭐⭐⭐⭐
**功能描述**：
- 用户可以将搭建好的工作流保存为模板
- 模板包含完整的节点配置和连接关系
- 支持模板命名、描述、分类管理

**实际应用场景**：
1. **标准流程固化**：HR部门将"员工入职流程"保存为模板，每次新员工入职时直接使用，无需重新搭建
2. **最佳实践共享**：销售团队将"合同审批流程"保存为模板，其他团队可以直接复用
3. **流程版本管理**：将不同版本的流程保存为不同模板，便于对比和回退

**带来的价值**：
- ✅ **效率提升**：工作流搭建时间从2小时缩短到5分钟（直接加载模板）
- ✅ **标准化**：确保团队使用统一的流程标准，减少人为错误
- ✅ **知识沉淀**：将优秀流程经验固化为模板，新员工可以快速上手

**技术实现**：新增WorkflowTemplate数据模型，提供保存/加载API

---

### 功能3：工作流模板审批发布 ⭐⭐⭐
**功能描述**：
- 模板创建后需要提交审批，只有通过审批的模板才能被其他用户使用
- 支持模板状态管理：草稿 → 已提交 → 已发布
- 超级管理员可以审批/拒绝模板发布

**实际应用场景**：
1. **质量管控**：确保只有经过验证的流程模板才能被广泛使用，避免错误流程影响业务
2. **合规要求**：金融行业要求所有业务流程必须经过审批才能上线
3. **权限管理**：防止未经授权的流程模板被误用，确保流程符合公司规范

**带来的价值**：
- ✅ **风险控制**：避免错误流程影响生产环境，降低业务风险
- ✅ **合规保障**：满足企业内控和审计要求
- ✅ **标准化管理**：确保所有发布的模板都经过严格审核

**技术实现**：参考App审批机制，实现状态流转和权限控制

---

### 功能4：工作流模板复用执行 ⭐⭐⭐⭐
**功能描述**：
- 从已发布的模板创建新的工作流实例
- 支持在模板基础上修改节点配置（如更换文件、调整参数）
- 同一模板可以同时运行多个实例，每个实例使用不同的配置

**实际应用场景**：
1. **批量处理不同数据**：使用"文档审核模板"，每次执行时上传不同的文档，系统自动处理
2. **个性化配置**：使用"合同审批模板"，但每次执行时指定不同的审批人
3. **A/B测试**：使用同一模板但不同配置，对比不同参数下的执行效果

**带来的价值**：
- ✅ **灵活性**：模板复用但不失灵活性，可以根据实际情况调整配置
- ✅ **效率提升**：无需重复搭建，只需调整配置即可执行
- ✅ **规模化**：支持大规模并发执行，提升业务处理能力

**技术实现**：支持template_id + app_node_configs参数，实现配置覆盖机制

```
工作流：爬取供应商网站价格 → 数据转换 → 审批
→ 系统自动爬取最新价格信息
→ 无需人工访问网站，自动获取数据
→ 提升数据采集效率
```

**场景2：审批结果通知**
```
工作流：文档审核 → 审批 → 发送邮件通知
→ 审批完成后自动发送邮件给相关人员
→ 无需人工发送通知，提升效率
→ 确保所有相关人员及时收到通知
```

**场景3：定时监控**
```
工作流：定时触发 → 爬取监控网站 → 条件判断 → 发送告警邮件
→ 系统定时检查网站状态
→ 发现异常时自动发送告警邮件
→ 实现7x24小时自动监控
```

**带来的价值**：
- ✅ **自动化**：减少人工操作，提升工作效率
- ✅ **实时性**：自动获取最新数据，无需等待
- ✅ **可靠性**：系统自动执行，减少人为错误
- ✅ **扩展性**：支持更多自动化场景

**技术实现**：集成现有爬虫API（`/crawler/start`）和邮件API（`send_email()`）

### 功能6：HITL审批逻辑增强 ⭐⭐⭐⭐
**功能描述**：
- 支持三种审批逻辑：any（任意一人）、all（全部通过）、majority（多数投票）
- 根据业务需求选择合适的审批方式
- 实时显示审批进度

**实际应用场景**：

**场景1：紧急审批（any逻辑）**
```
场景：紧急采购需要快速审批
→ 任意一个审批人同意即可通过
→ 提升审批速度，适合紧急场景
→ 当前已支持
```

**场景2：重要决策（all逻辑）**
```
场景：重大合同需要所有部门负责人审批
→ 必须所有审批人都同意才能通过
→ 任何一个人拒绝则整体拒绝
→ 确保重要决策经过充分讨论
```

**场景3：民主决策（majority逻辑）**
```
场景：供应商选择需要多个部门投票
→ 多数人同意则通过，多数人拒绝则拒绝
→ 处理平票情况（可配置默认行为）
→ 适合需要集体决策的场景
```

**带来的价值**：
- ✅ **灵活性**：根据业务场景选择合适的审批方式
- ✅ **合规性**：满足不同业务场景的审批要求
- ✅ **透明度**：实时显示审批进度，清晰了解审批状态
- ✅ **效率**：不同场景使用不同逻辑，平衡效率和风险

**技术实现**：扩展`approvalLogic`字段，支持'any'/'all'/'majority'三种值

---

### 功能7：Condition Node增强（通俗化表达） ⭐⭐⭐⭐
**功能描述**：
- 支持两种条件配置方式：
  - 方式一：用通俗的文字替换操作符（如"供应商是否符合条件"等同于==）
  - 方式二：使用标准操作符（equals, not equals等）- 已实现
- 条件输出不仅包含true/false，还包含业务含义（如"True，代表有符合条件的供应商"）

**实际应用场景**：

**场景1：非技术人员使用**
```
当前方式：has_qualified_suppliers === true
→ 非技术人员看不懂，容易出错
→ 需要技术人员协助配置

新方式：供应商是否符合条件
→ 一目了然，非技术人员也能理解
→ 降低使用门槛，提升易用性
```

**场景2：业务含义清晰**
```
当前输出：True
→ 只知道结果是true，不知道代表什么
→ 需要查看条件表达式才能理解

新输出：True，代表有符合条件的供应商
→ 直接显示业务含义，无需查看表达式
→ 提升可读性，降低理解成本
```

**场景3：复杂条件组合**
```
条件：供应商是否符合条件 AND 供应商数量是否大于0
输出：True，代表有符合条件的供应商，且供应商数量为3个
→ 不仅显示结果，还显示关键数据
→ 帮助用户理解条件判断的依据
```

**带来的价值**：
- ✅ **易用性**：非技术人员也能轻松使用，降低使用门槛
- ✅ **可读性**：输出包含业务含义，更容易理解
- ✅ **专业性**：既支持通俗表达，也支持专业操作符
- ✅ **灵活性**：用户可以根据需要选择配置方式

**技术实现**：
- 前端：提供文字选择界面，映射到操作符
- 后端：解析文字表达式，转换为标准表达式
- 输出：增加业务含义字段，根据条件自动生成

**测试要点**：
- 验证多个工作流并发执行
- 验证审批后正确恢复

```
工作流：审批 → 发邮件通知 → 结束
```
