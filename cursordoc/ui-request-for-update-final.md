# Request for Update Node Output UI 最终设计方案

## 设计原则

1. **避免信息重复**：提交更新信息的用户和上传文件的用户通常是同一个人，因此只在 Update Information 区域显示用户信息
2. **统一术语**：统一使用 "Updated By" 和 "Update Time"
3. **清晰的信息层次**：文件元信息只显示文件相关属性，不重复显示用户信息

## 8. 完整交互流程

### 8.1 初始加载

```
时间线：
T0: 用户打开 Node Output 面板
    - Update Information 立即显示（数据已在 outputs 中）
    - Updated By 信息立即显示
    - Attached Files 区域显示 Loading Spinner ⏳

T1: 开始加载文件信息（200-1000ms）
    - 显示 "Loading file information..."
    - 或显示 "Loading file 1 of 2..."

T2: 文件信息加载完成
    - 显示文件卡片（文件名、大小、上传时间）
    - 文件内容默认折叠
    - 用户可以点击展开查看内容
```

### 8.2 用户交互

```
用户操作：
1. 点击 [▶ Show File Content] 
   → 展开文件内容
   → 按钮变为 [▼ Hide File Content]

2. 点击 [▼ Hide File Content]
   → 折叠文件内容
   → 按钮变为 [▶ Show File Content]

3. 多个文件可以独立展开/折叠
```

## 10. 设计总结

### 10.1 信息层次

```
Level 1: Update Information
  - update_info 内容
  - Updated By 信息（用户信息只显示一次）

Level 2: Attached Files
  - 文件元信息（文件名、大小、上传时间）
  - 文件内容（可折叠）
  - ❌ 不显示上传人信息（避免重复）
```

### 10.2 关键设计决策

1. ✅ **避免信息重复**：用户信息只在 Update Information 区域显示一次
2. ✅ **统一术语**：使用 "Updated By" 和 "Update Time"
3. ✅ **清晰的信息层次**：文件卡片只显示文件相关属性
4. ✅ **良好的用户体验**：Loading Spinner 提供视觉反馈
5. ✅ **灵活的内容展示**：文件内容可折叠/展开

### 10.3 推荐方案组合

- **文件元信息布局**：卡片形式，带图标（方案C）
- **文件内容格式化**：智能格式化，自动转换为 Markdown（方案3）
- **加载状态**：Loading Spinner（方案B）
- **术语统一**：Updated By / Update Time

## 11. 与 Approval 类型的对比

### Approval 类型显示：

```
Reviewed By:
• Email: reviewer@example.com
• Job Title: Operations Manager
• Decision Time: 2026/1/13 16:16:22
```

### Request for Update 类型显示：

```
Updated By:
• Email: submitter@example.com
• Job Title: Operations Manager
• Update Time: 2026/1/13 16:16:22
```

**区别**：
- Approval: "Reviewed By" + "Decision Time"（审核决定）
- Update: "Updated By" + "Update Time"（提交更新）
