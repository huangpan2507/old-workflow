# Workflow完整解决方案指南

## 文档概述

本文档从**Workflow开发者**和**Workflow使用者**两个角度，全面呈现Workflow系统的功能、使用场景和价值。帮助客户理解Workflow如何支持业务流程自动化，以及如何从搭建到使用的完整流程。

---

## 第一部分：Workflow开发者视角

### 一、Workflow生命周期管理

#### 1.1 创建Workflow（Create）

**功能描述**：
开发者可以在Workbench中通过拖拽方式创建Workflow，配置节点和连接关系。

**核心能力**：
- 可视化拖拽界面，无需编写代码
- 支持6种节点类型（Start、App、Condition、Human in the Loop、Tool、End）
- 灵活配置节点参数和连接关系
- 实时预览Workflow结构

**使用场景**：
1. **业务流程自动化**：将重复的业务流程固化为Workflow
   - 示例：供应商审核流程、合同审批流程、文档分析流程
2. **AI任务编排**：将多个AI应用串联成完整任务链
   - 示例：文档上传 → AI分析 → 数据提取 → 条件判断 → 审批 → 邮件通知
3. **复杂决策流程**：实现多分支、多条件的业务逻辑
   - 示例：根据分析结果自动选择不同的处理路径

**业务价值**：
- ✅ **效率提升**：将手动流程自动化，节省80%以上时间
- ✅ **标准化**：确保流程执行的一致性，减少人为错误
- ✅ **可追溯**：完整记录执行过程，便于审计和优化

---

#### 1.2 保存Workflow（Save）

**功能描述**：
开发者可以将搭建好的Workflow保存为模板，供后续复用。

**核心能力**：
- 保存Workflow配置（节点、连接、参数）
- 支持命名、描述、分类管理
- 支持版本管理（后续功能）
- 支持草稿保存，随时继续编辑

**使用场景**：
1. **标准流程固化**：将经过验证的流程保存为模板
   - 示例：HR部门将"员工入职流程"保存为模板，每次新员工入职时直接使用
2. **最佳实践共享**：团队将优秀流程保存为模板，供其他团队复用
   - 示例：销售团队将"合同审批流程"保存为模板，其他团队可以直接使用
3. **流程版本管理**：将不同版本的流程保存为不同模板，便于对比和回退
   - 示例：保存"合同审批流程v1.0"和"合同审批流程v2.0"

**业务价值**：
- ✅ **复用效率**：工作流搭建时间从2小时缩短到5分钟（直接加载模板）
- ✅ **知识沉淀**：将优秀流程经验固化为模板，新员工可以快速上手
- ✅ **标准化管理**：确保团队使用统一的流程标准

---

#### 1.3 发布Workflow（Publish）

**功能描述**：
开发者可以将保存的Workflow提交审批，审批通过后发布，供其他用户使用。

**核心能力**：
- 提交审批：将Workflow从"草稿"状态提交为"待审批"
- 审批流程：超级管理员审批通过后，Workflow变为"已发布"状态
- 状态管理：draft（草稿）→ submitted（已提交）→ published（已发布）
- 权限控制：只有已发布的Workflow才能被其他用户使用

**使用场景**：
1. **质量管控**：确保只有经过验证的流程才能被广泛使用
   - 示例：财务部门创建的"报销审批流程"必须经过管理员审批才能发布
2. **合规要求**：满足企业内控和审计要求
   - 示例：金融行业要求所有业务流程必须经过审批才能上线
3. **权限管理**：防止未经授权的流程被误用
   - 示例：只有经过审批的Workflow才能被其他部门使用

**业务价值**：
- ✅ **风险控制**：避免错误流程影响生产环境，降低业务风险
- ✅ **合规保障**：满足企业内控和审计要求
- ✅ **标准化管理**：确保所有发布的Workflow都经过严格审核

**与Application发布的关系**：
- Application创建后也需要发布，发布时也要审核
- Workflow和Application使用相同的审批发布机制
- 确保整个平台的内容质量

---

#### 1.4 审核Workflow（Review）

**功能描述**：
超级管理员可以审核开发者提交的Workflow，决定是否批准发布。

**核心能力**：
- 查看待审核的Workflow列表
- 查看Workflow详细配置（节点、连接、参数）
- 批准或拒绝Workflow发布
- 提供拒绝原因，帮助开发者改进

**使用场景**：
1. **质量把关**：管理员审核Workflow是否符合公司规范
   - 示例：检查审批流程是否包含必要的审批节点
2. **安全审查**：确保Workflow不会泄露敏感信息
   - 示例：检查是否包含不安全的操作（如删除数据）
3. **合规检查**：确保Workflow符合行业法规要求
   - 示例：金融行业需要确保审批流程符合监管要求

**业务价值**：
- ✅ **质量保障**：确保发布的Workflow都经过专业审核
- ✅ **风险防范**：及时发现和阻止有问题的Workflow
- ✅ **持续改进**：通过反馈帮助开发者改进Workflow质量

---

#### 1.5 分享Workflow（Share）

**功能描述**：
开发者可以将已发布的Workflow分享给其他用户或团队使用。

**核心能力**：
- 分享链接：生成Workflow分享链接
- 权限控制：控制谁可以使用该Workflow
- 使用统计：查看Workflow被使用的次数和用户
- 反馈收集：收集使用者反馈，持续优化

**使用场景**：
1. **跨部门协作**：将Workflow分享给其他部门使用
   - 示例：IT部门创建的"系统部署流程"分享给运维团队
2. **知识共享**：将优秀实践分享给全公司
   - 示例：销售团队将"客户跟进流程"分享给所有销售团队
3. **外部合作**：与合作伙伴分享标准流程
   - 示例：将"供应商审核流程"分享给合作伙伴

**业务价值**：
- ✅ **协作效率**：促进跨部门协作，减少重复工作
- ✅ **知识传播**：将优秀实践快速传播到全公司
- ✅ **生态建设**：建立Workflow共享生态，提升整体效率

---

#### 1.6 多实例运行（Multi-Instance Execution）

**功能描述**：
同一个Workflow模板可以同时运行多个实例，每个实例使用不同的配置和数据。

**核心能力**：
- 从模板创建实例：基于已发布的Workflow模板创建新的执行实例
- 配置覆盖：每个实例可以有不同的配置（如不同的文件、不同的参数）
- 并发执行：多个实例可以同时运行，互不干扰
- 独立追踪：每个实例的执行状态和结果独立追踪

**使用场景**：
1. **批量处理不同数据**：使用同一个模板处理不同的数据
   - 示例：使用"文档审核模板"，每次执行时上传不同的文档
   - 场景：HR部门需要审核100份简历，使用同一个模板，每次上传不同的简历文件
2. **个性化配置**：使用同一模板但不同的配置参数
   - 示例：使用"合同审批模板"，但每次执行时指定不同的审批人
   - 场景：不同地区的合同需要不同的审批人，但流程相同
3. **A/B测试**：使用同一模板但不同配置，对比效果
   - 示例：使用"客户跟进模板"，但使用不同的AI模型参数，对比效果
   - 场景：测试不同AI模型在客户跟进中的表现

**业务价值**：
- ✅ **规模化处理**：支持大规模并发执行，提升业务处理能力
- ✅ **灵活性**：模板复用但不失灵活性，可以根据实际情况调整配置
- ✅ **效率提升**：无需重复搭建，只需调整配置即可执行

**技术实现**：
- 每个实例有独立的`task_id`和`thread_id`
- 配置通过`app_node_configs`参数覆盖模板默认配置
- 支持文件上传覆盖（`fileDagRunMap`）

---

#### 1.7 异步运行（Async Execution）

**功能描述**：
Workflow支持异步执行，不会阻塞用户操作，特别适合长时间运行的任务。

**核心能力**：
- 异步启动：提交Workflow后立即返回，不等待执行完成
- 状态追踪：通过WebSocket实时推送执行状态
- 中断恢复：支持人工审批等中断场景，审批完成后自动恢复
- 资源释放：等待期间不占用服务器资源，支持高并发

**使用场景**：
1. **长时间任务**：处理需要较长时间的任务
   - 示例：文档分析、数据爬取、批量处理
   - 场景：分析100页PDF文档，需要10分钟，异步执行不阻塞用户
2. **人工审批**：需要人工审批的任务
   - 示例：合同审批、费用报销、供应商审核
   - 场景：审批人可能不在线，系统等待审批，不占用资源
3. **高并发场景**：同时处理多个任务
   - 示例：销售团队同时提交50个合同审批流程
   - 场景：系统可以并行处理，不会因为资源阻塞导致卡顿

**业务价值**：
- ✅ **用户体验**：用户提交后可以继续其他工作，不等待
- ✅ **系统性能**：支持高并发，服务器资源利用率提升80%
- ✅ **可扩展性**：支持10倍以上的并发Workflow

**技术实现**：
- 使用LangGraph的interrupt机制，支持异步中断和恢复
- 使用WebSocket实时推送执行状态
- Checkpoint自动管理，支持状态恢复

### 二、Workflow节点组件详解

Workflow由6种节点类型组成，每种节点都有特定的功能和作用。

#### 2.1 Start Node（起始节点）🟢

**功能描述**：
Workflow的入口节点，标记Workflow开始执行的位置。

**核心能力**：
- 接收初始输入数据（`input_data`）
- 将输入数据传递给下游节点
- 标记Workflow执行开始

**使用场景**：
1. **接收用户输入**：接收用户上传的文件、输入的参数等
   - 示例：接收用户上传的合同文档、输入的审核要求
2. **初始化数据**：为Workflow提供初始数据
   - 示例：提供供应商名称、合同金额等初始信息

**输出格式**：
```json
{
  "status": "completed",
  "output": {
    // 直接输出接收到的input_data的所有字段
    "document_url": "https://example.com/doc.pdf",
    "analysis_type": "supplier_check"
  }
}
```

**注意事项**：
- 每个Workflow必须有且仅有一个Start Node
- Start Node必须是Workflow的第一个节点

---

#### 2.2 App Node（应用节点）🔵

**功能描述**：
执行AI分析任务，调用已创建的Application进行智能处理。

**核心能力**：
- 拖拽Application到Workflow中，自动封装为App Node
- 执行AI分析、文档处理、数据提取等任务
- 支持配置参数（如提示词、文件、要求等）
- 支持多实例运行时的配置覆盖

**使用场景**：
1. **文档分析**：分析合同、报告、政策等文档
   - 示例：分析合同条款，提取关键信息
2. **数据提取**：从非结构化数据中提取结构化信息
   - 示例：从简历中提取姓名、学历、工作经验等信息
3. **智能判断**：使用AI进行智能判断和分类
   - 示例：判断供应商是否符合资质要求

**配置方式**：
- **开发时**：在Workflow中配置App Node的参数（提示词、文件等）
- **运行时**：使用者可以覆盖配置（上传不同的文件、修改参数）

**输出格式**：
```json
{
  "status": "completed",
  "output": {
    "content": "AI分析结果文本",
    "structured_data": {
      // 提取的结构化数据
    }
  }
}
```

**与Application的关系**：
- App Node是对Application的封装
- Application需要先创建并发布，才能在Workflow中使用
- 一个Application可以被多个Workflow使用

---

#### 2.3 Condition Node（条件节点）🟠

**功能描述**：
根据条件判断结果，决定Workflow的执行路径（分支）。

**核心能力**：
- 支持多种条件表达式（等于、不等于、大于、小于等）
- 支持逻辑运算符（AND、OR、NOT）
- 根据条件结果路由到不同的下游节点
- 输出条件判断结果和业务含义

**使用场景**：
1. **分支决策**：根据分析结果选择不同的处理路径
   - 示例：如果供应商符合条件，走快速审批；否则走详细审核
2. **数据验证**：验证数据是否符合要求
   - 示例：检查合同金额是否超过阈值，超过需要特殊审批
3. **状态判断**：根据当前状态决定下一步操作
   - 示例：如果文档已审核，发送通知；否则继续审核

**条件配置示例**：
```javascript
// 方式一：标准操作符（当前支持）
condition: "supplier_score > 80 AND supplier_status === 'active'"

// 方式二：通俗文字（后续支持）
condition: "供应商评分是否大于80分 AND 供应商状态是否为活跃"
```

**输出格式**：
```json
{
  "status": "completed",
  "output": {
    "condition_result": true,
    "business_meaning": "供应商符合条件，且评分大于80分",
    "route": "approved_path"  // 路由到哪个下游节点
  }
}
```

**注意事项**：
- Condition Node必须有至少两个下游节点（true路径和false路径）
- 条件表达式必须返回布尔值

---

#### 2.4 Human in the Loop Node（人工审批节点）🟡

**功能描述**：
暂停Workflow执行，等待人工审批或确认，审批完成后自动继续执行。

**核心能力**：
- 创建审批任务，发送给指定的审批人
- 支持多种审批逻辑：any（任意一人）、all（全部通过）、majority（多数投票）
- 异步等待，不阻塞服务器资源
- 审批完成后自动恢复Workflow执行

**使用场景**：
1. **重要决策审批**：需要人工确认的重要决策
   - 示例：合同审批、费用报销、供应商审核
   - 场景：合同金额超过100万，需要财务总监审批
2. **质量把关**：确保关键环节有人工审核
   - 示例：AI分析结果需要人工确认准确性
   - 场景：AI提取的供应商信息需要人工核实
3. **合规要求**：满足企业合规和审计要求
   - 示例：金融行业要求所有交易必须经过审批
   - 场景：所有超过10万的交易都需要审批

**审批逻辑**：
- **any模式**：任意一个审批人同意即可通过（适合紧急场景）
- **all模式**：必须所有审批人都同意才能通过（适合重要决策）
- **majority模式**：多数人决定结果（适合民主决策）

**配置示例**：
```json
{
  "taskType": "approval",
  "approvalConfig": {
    "approverJobTitleIds": [1, 2, 3],  // 审批人职位ID
    "approvalLogic": "all",  // 审批逻辑：any/all/majority
    "title": "合同审批",
    "description": "请审批以下合同"
  }
}
```

**异步特性**：
- Workflow执行到HITL节点时，系统暂停执行并释放资源
- 审批人可以在任意时间完成审批
- 审批完成后，系统自动恢复Workflow执行
- 支持多个Workflow同时等待审批，互不干扰

**输出格式**：
```json
{
  "status": "completed",
  "output": {
    "decision": "approve",  // approve/reject
    "reviewer_info": {
      "reviewer_id": "user123",
      "reviewer_name": "张三",
      "review_time": "2024-01-15T10:30:00Z"
    },
    "reason": "审批通过，合同条款符合要求"
  }
}
```

#### 2.5 Tool Node（工具节点）⚙️

**功能描述**：
执行外部工具操作，如发送邮件、爬取网页、调用API等。

**核心能力**：
- **邮件发送**：发送通知邮件，支持附件
- **网页爬虫**：爬取网页内容，提取数据
- **MCP工具集成**：通过MCP方式集成Microsoft Graph API
  - SharePoint：访问和操作SharePoint文档
  - Teams：发送Teams消息、创建会议
  - OneDrive：访问和操作OneDrive文件

**使用场景**：

**场景1：邮件通知**
```
工作流：文档审核 → 审批 → 发送邮件通知 → 结束
→ 审批完成后，自动发送邮件给相关人员
→ 邮件内容包含审批结果和详细信息
→ 无需人工发送通知，提升效率
```

**场景2：数据爬取**
```
工作流：文档分析 → 提取供应商名称 → 爬虫获取供应商详情 → 审批
→ 系统自动爬取供应商官网信息
→ 将爬取结果传递给审批节点
→ 实现自动化数据采集
```

**场景3：SharePoint集成**
```
工作流：文档分析 → 提取关键信息 → 保存到SharePoint → 发送Teams通知
→ 自动将分析结果保存到SharePoint文档库
→ 在Teams中通知相关人员查看
→ 实现与Microsoft 365的深度集成
```

**场景4：定时监控**
```
工作流：定时触发 → 爬取监控网站 → 条件判断 → 发送告警邮件
→ 系统定时检查网站状态
→ 发现异常时自动发送告警邮件
→ 实现7x24小时自动监控
```

**邮件配置示例**：
```json
{
  "toolType": "email",
  "config": {
    "to": ["user1@example.com", "user2@example.com"],
    "subject": "合同审批结果通知",
    "body": "合同已审批通过，请查看附件",
    "attachments": ["contract.pdf"]
  }
}
```

**爬虫配置示例**：
```json
{
  "toolType": "crawler",
  "config": {
    "url": "https://supplier.com/info",
    "method": "GET",
    "selectors": {
      "company_name": ".company-name",
      "contact": ".contact-info"
    }
  }
}
```

**MCP工具配置示例**：
```json
{
  "toolType": "mcp_graph_api",
  "config": {
    "service": "sharepoint",
    "action": "upload_file",
    "site_id": "site123",
    "file_path": "/documents/analysis_result.pdf"
  }
}
```

**输出格式**：
```json
{
  "status": "completed",
  "output": {
    "tool_type": "email",
    "result": {
      "success": true,
      "message_id": "msg123",
      "sent_at": "2024-01-15T10:30:00Z"
    }
  }
}
```

**注意事项**：
- Tool Node需要配置相应的工具参数
- 邮件和爬虫功能需要团队已准备好的接口
- MCP工具需要配置Microsoft Graph API认证

---

#### 2.6 End Node（结束节点）🔴

**功能描述**：
Workflow的出口节点，标记Workflow执行完成。

**核心能力**：
- 标记Workflow执行结束
- 汇总最终执行结果
- 触发完成通知（可选）

**使用场景**：
1. **正常结束**：Workflow正常执行完成
2. **结果汇总**：汇总所有节点的执行结果
3. **完成通知**：发送Workflow执行完成的通知

**输出格式**：
```json
{
  "status": "completed",
  "output": {
    "workflow_result": "success",
    "summary": {
      "total_nodes": 6,
      "completed_nodes": 6,
      "execution_time": "5m30s"
    },
    "final_data": {
      // 最终的执行结果数据
    }
  }
}
```

**注意事项**：
- 每个Workflow必须有且仅有一个End Node
- End Node必须是Workflow的最后一个节点

## 第四部分：技术架构说明

### 一、Workflow执行引擎

**技术栈**：
- **LangGraph**：Workflow执行引擎
- **LangChain**：AI应用集成
- **PostgreSQL**：状态持久化
- **WebSocket**：实时状态推送

**核心机制**：
- **Checkpoint机制**：自动保存执行状态，支持中断和恢复
- **Interrupt机制**：支持异步中断（如人工审批），不阻塞执行
- **状态管理**：完整的状态追踪和恢复机制

---

### 二、节点执行机制

**节点类型**：
- **同步节点**：Start、App、Condition、Tool、End（立即执行）
- **异步节点**：Human in the Loop（等待人工审批）

**执行流程**：
```
1. 节点执行 → 2. 保存Checkpoint → 3. 继续下一个节点
                ↓
            (如果是HITL节点)
                ↓
        4. 触发Interrupt → 5. 等待审批 → 6. 恢复执行
```

---

### 三、多实例运行机制

**实例隔离**：
- 每个实例有独立的`task_id`和`thread_id`
- 独立的Checkpoint存储
- 独立的执行状态追踪

**配置覆盖**：
- 模板定义默认配置
- 实例执行时通过`app_node_configs`覆盖
- 支持部分覆盖（只覆盖指定的配置项）
